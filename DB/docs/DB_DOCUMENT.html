1. 전체 구조 개요

ELAW DB는 4개 레이어로 구성됩니다. 각 레이어는 독립적인 모델 파일로 관리되며, models_register.py를 통해 Django ORM에 일괄 등록됩니다.

레이어 1 기본 도메인 — models.py
사용자, 기업, 학습 목표, 커리큘럼, 풀이 이력, 채용, 포트폴리오 등 플랫폼 핵심 12개 테이블
레이어 2 데이터셋 — models_dataset.py
HuggingFace 이력서·채용공고 원본 데이터셋 저장 및 AI 파싱 결과 5개 테이블
레이어 3 문제 — models_problems.py
직군별 6,000개 문제, 클러스터, 선수과목 그래프, 학습 경로 메타 5개 테이블
레이어 4 추천·갭·포트폴리오 — models_new.py
스킬 갭 분석, 문제 추천, 포트폴리오 버전 관리, RLHF 피드백 4개 테이블

파일 구조

backend/core/
├── models.py              # 레이어 1 (기본 도메인 12개 모델)
├── models_dataset.py      # 레이어 2 (데이터셋 5개 모델)
├── models_problems.py     # 레이어 3 (문제 5개 모델)
├── models_new.py          # 레이어 4 (추천·갭·포트폴리오 4개 모델)
├── models_register.py     # 전체 모델 Django 앱 등록 허브
├── migrations/
│   ├── 0001_initial.py              # 기본 12개 테이블
│   ├── 0002_user_is_staff_...py     # User 필드 추가
│   ├── 0003_dataset_tables.py       # 데이터셋 5개 테이블
│   ├── 0004_merge_20260421.py       # 병합

    └── sync_platforms.py  # 백준·GitHub API 동기화

AI가 생성한 포트폴리오의 각 버전. is_final=True인 버전이 사용자에게 표시됩니다.

필드타입설명
portfolio_idForeignKeyFK→portfolios
user_idForeignKeyFK→users
posting_idForeignKeyFK→job_postings (nullable)
structured_resume / skill_analysis / problem_historyJSONField생성 당시 입력 데이터 스냅샷
generated_contentJSONField섹션별 생성 결과
generation_methodCharField(20)gemini / model_v1 / model_v2 / manual
generation_promptTextField재현·개선용 프롬프트 보관
versionPositiveSmallIntegerField버전 번호
is_finalBooleanField현재 표시 버전 여부
quality_scoreFloatField자동 품질 평가 점수

5-4. portfolio_feedback (RLHF 피드백)

사용자가 포트폴리오 스냅샷에 남긴 피드백. used_for_training=True 데이터로 모델을 재학습합니다.

필드타입설명
snapshot_idForeignKeyFK→portfolio_snapshots
user_idForeignKeyFK→users
ratingIntegerField1(매우불만족) ~ 5(매우만족)
feedback_textTextField자유 텍스트
edited_contentJSONField사용자 수정 내용 (원본 비교용)
rlhf_labelsJSONField{overall, relevance, completeness, preferred_sections, edit_distance}
used_for_trainingBooleanField모델 재학습 사용 여부
training_batch_idCharField(100)학습 배치 식별자

유니크: (snapshot, user)

↓
3

7-2. 학습 분석 흐름

1
sync_platforms → 백준·GitHub API → solve_history 동기화
↓
2
solve_history 집계 → learning_stats (언어별·태그별 정답률) 갱신
↓
3
learning_stats + job_postings.required_skills 비교 → skill_gaps 생성
↓
4
skill_gaps + problem_edges (선수과목) → problem_recommendations 생성

7-3. 포트폴리오 생성 및 RLHF 흐름

1
dataset_resumes(구조화 이력서) + skill_gaps + solve_history 요약 → Gemini API 입력
↓
2
Gemini 응답 → portfolio_snapshots 저장 (v1, is_final=True)
↓
3
사용자 피드백 → portfolio_feedback 저장 (rlhf_labels 포함)
↓
4
used_for_training=True 데이터 축적 → 자체 모델 재학습 (model_v1 → model_v2)

7-4. 매칭 흐름

1
dataset_entries (HuggingFace) → Gemini 파싱 → dataset_resumes + dataset_job_descriptions
↓
2
이력서·공고 쌍 매칭 → dataset_match_scores (AI 학습 레이블)
↓
3
실사용자 portfolios + 기업 job_postings → AI 매칭 → matches

11. 환경 설정

현재 (개발)

DATABASES = {
    'default': {
        'ENGINE': 'django.db.backends.sqlite3',
        'NAME': BASE_DIR / 'db.sqlite3',
    }
}

운영 전환 시 (MySQL)

DATABASES = {
    'default': {
        'ENGINE': 'django.db.backends.mysql',
        'NAME': os.getenv('DB_NAME'),
        'USER': os.getenv('DB_USER'),
        'PASSWORD': os.getenv('DB_PASSWORD'),
        'HOST': os.getenv('DB_HOST', 'localhost'),
        'PORT': os.getenv('DB_PORT', '3306'),
        'OPTIONS': {'charset': 'utf8mb4'},
    }
}

필수 환경변수 (.env)

변수명용도필수
GEMINI_API_KEY커리큘럼·포트폴리오·데이터셋 파싱 AI운영 필수
GITHUB_TOKENGitHub API ETL (sync_platforms)선택
DB_NAME / DB_USER / DB_PASSWORD / DB_HOSTMySQL 연결 정보MySQL 전환 시
SECRET_KEYDjango 보안 키운영 필수
운영 배포 전 필수 조치: SECRET_KEY를 50자 이상 랜덤 값으로 교체, DEBUG=False 설정, ALLOWED_HOSTS에 도메인 명시, CORS_ALLOW_ALL_ORIGINS=False 설정.
