## Git Commit 작성 규칙

### 작성 형식

커밋 메시지는 다음 형식을 따릅니다.

타입: 변경 내용

예시

* feat: 로그인 기능 추가
* fix: 로그인 오류 수정
* docs: Git 규칙 문서 추가

### 타입 의미

* **feat** : 새로운 기능 추가
* **fix** : 버그 수정
* **docs** : 문서 수정
* **style** : 코드 포맷팅, 세미콜론 누락 등 코드 동작과 관련 없는 수정
* **refactor** : 코드 리팩터링 (기능 변경 없음)
* **test** : 테스트 코드 추가 또는 테스트 코드 리팩터링 (프로덕션 코드 변경 없음)
* **chore** : 빌드 설정, 패키지 매니저 수정 등 기타 작업
* **design** : CSS 등 사용자 UI 디자인 변경
* **comment** : 필요한 주석 추가 및 변경
* **rename** : 파일 또는 폴더 이름 변경
* **remove** : 파일 삭제
* **!BREAKING CHANGE** : 기존 API와 호환되지 않는 큰 변경
* **!HOTFIX** : 긴급하게 치명적인 버그를 수정하는 경우

### Description (설명란)

* 커밋의 변경 내용을 가능한 한 **구체적으로 작성합니다.**
* 변경 사항이 많을 경우 **핵심적인 내용만 간단히 요약합니다.**
* 변경 내용이 많아지기 전에 **작업 단위로 나누어 자주 커밋하는 것이 좋습니다.**

이를 통해 문제 발생 시 변경 이력을 쉽게 추적할 수 있으며,
프로젝트의 변경 내용을 팀원들이 빠르게 이해할 수 있습니다.
